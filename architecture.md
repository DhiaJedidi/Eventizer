# architecture.md ‚Äî Eventizer Landing Page

> **v1 stack: Next.js (App Router) + Payload CMS 3.** Payload installs *into* the same
> Next.js app ‚Äî one codebase, one deploy. The public landing page and the Payload admin panel
> share the same `app/` directory via route groups. Content (case studies, team, section copy)
> is editable in the admin panel; contact-form leads are stored as a Payload collection.

---

## Stack

| Layer | Choice | Reason |
|---|---|---|
| Framework | Next.js 14+ (App Router) | Required by Payload 3; SSG/SSR, `next/image`, SEO-friendly output |
| CMS / backend | **Payload CMS 3** | Lives inside the Next.js app. Admin panel, content collections, auth, Local API ‚Äî no separate server |
| Language | TypeScript | Type safety across components, props, and Payload's generated `payload-types.ts` |
| Database | **SQLite** (`@payloadcms/db-sqlite`, Drizzle) | Lightest data layer ‚Äî a single file. Local file in dev; **Turso/libSQL** in production (managed SQLite, no DB server to run). See "Why a database at all?" below |
| Styling | Tailwind CSS + CSS custom properties | Utility-first; design tokens from `design.md` / `tokens.json` |
| Forms | React Hook Form + Zod | Validation without re-renders; Zod schema shared with the API route |
| Media storage | Local (dev) ‚Üí `@payloadcms/storage-vercel-blob` (prod) | Uploaded images (case studies, team) need durable storage; Vercel's filesystem is ephemeral |
| Email delivery | Resend | Transactional email on form submit (in addition to storing the lead) |
| Analytics | Google Analytics 4 | Conversion event tracking |
| Hosting | Vercel | Runs the Next.js + Payload app as serverless functions; Turso + Vercel Blob are serverless-compatible |

> **Why a database at all?** Payload is a CMS ‚Äî editable content and form submissions *must*
> be stored somewhere. There is no Payload-without-a-database mode. SQLite is the lightest
> option: no DB server to provision, just a file. In production on Vercel the filesystem is
> ephemeral, so the SQLite file lives on **Turso/libSQL** (managed, free tier). If you would
> rather not run any database service at all, the alternative is to drop Payload and hardcode
> all copy from `copy.md` ‚Äî but that removes "editable content," which was requested.

> **Note for [performance.md](performance.md):** `output: 'export'` (static export) is **not
> possible** with Payload ‚Äî the admin panel and API need a server runtime. Deploy as Vercel
> serverless (`output` default / `'standalone'`). The contact API and admin run server-side.

## How Payload sits inside Next.js

Payload 3 adds two route groups to `app/`:

- **`app/(frontend)/`** ‚Äî the public Eventizer landing page (what visitors see). This is where
  all the section components render.
- **`app/(payload)/`** ‚Äî Payload's admin UI (`/admin`) and REST/GraphQL API (`/api/...`),
  largely generated by `create-payload-app`. You rarely edit these files by hand.

Both groups share the same root URL. Visitors hit `/`; editors hit `/admin`.

The frontend reads content through Payload's **Local API** (direct function calls, no HTTP) in
Server Components ‚Äî fast, no extra network hop.

## Content model (Payload)

Each section maps to a collection or global. Field copy/defaults come from `copy.md`.

| Section | Payload type | Key fields |
|---|---|---|
| Hero | Global `Hero` | eyebrow, h1, subheadline, primaryCta, secondaryCta, trustStrip, heroImageAlt |
| Stats | Global `Stats` | `metrics[]` ‚Üí { value, label } (√ó4) |
| Pillars | *static* | Fixed in `Pillars.tsx` from `copy.md` (no admin editing in v1) |
| Platform | Global `Platform` | eyebrow, h2, body, `features[]`, proofPoint, cta, screenshotAlt |
| Case Studies | Collection `CaseStudies` | title, badges[], keyStat, statLabel, description, photo (‚Üí Media), alt, order |
| Team | Collection `TeamMembers` | name, role, oneLiner, headshot (‚Üí Media), alt, order |
| Why Eventizer | Global `WhyEventizer` | `differentiators[]` ‚Üí { title, body } (√ó3) |
| Contact | Global `ContactInfo` + Collection `Submissions` | info: phone, email, whatsapp, hours ¬∑ submissions: leads |

**`Submissions` collection** (contact-form leads)
- Fields: `firstName`, `lastName`, `email`, `phone`, `eventType` (select), `message`,
  `createdAt`, `status` (new/contacted).
- `access`: `create` open to the API route only; `read/update` restricted to authenticated
  admins. Never publicly readable.
- Created by `POST /api/contact` via the Local API ‚Äî not by a public Payload REST call.

**`Media` collection**
- Upload-enabled. Each item carries an `alt` field (required ‚Äî enforces the
  [seo.md](seo.md) ¬ß8 alt-text rule). Payload generates resized variants; render through
  `next/image` for WebP + `srcset`.
- Dev: local disk. Prod: Vercel Blob via `@payloadcms/storage-vercel-blob`.

## Contact form flow

```
Client (Contact.tsx, React Hook Form + Zod)
   ‚îÇ  validate on blur (errors from copy.md)
   ‚ñº
POST /api/contact  (app/(frontend)/api/contact/route.ts)
   ‚îÇ  1. Re-validate with the SAME Zod schema (lib/schemas.ts) ‚Äî never trust the client
   ‚îÇ  2. payload.create({ collection: 'submissions', data })   ‚Üê store the lead
   ‚îÇ  3. sendEmail(data) via Resend                            ‚Üê notify the team
   ‚îÇ  4. return { ok: true }  (Cache-Control: no-store)
   ‚ñº
Client ‚Üí success state (replaces form; copy from copy.md). On failure ‚Üí system-error copy.
```

- The Zod schema in `lib/schemas.ts` is imported by both the client form and the route ‚Äî one
  source of validation truth.
- GA4 `contact_form_submit` fires on the success state ([seo.md](seo.md) ¬ß15).

## Performance requirements

| Target | Requirement |
|---|---|
| Hero LCP | < 2.5s ‚Äî `<Image priority />` on the static hero asset |
| CLS | < 0.1 ‚Äî reserve dimensions for all images |
| INP | < 200ms ‚Äî no blocking JS on main thread |
| Lighthouse Performance | ‚â• 90 |
| Lighthouse Accessibility | ‚â• 90 |
| Lighthouse SEO | ‚â• 95 |
| Lighthouse Best Practices | ‚â• 90 |

See [performance.md](performance.md) for full image/font/bundle/caching strategy. Note Payload's
admin bundle is code-split under `/admin` and **does not** ship with the public landing page ‚Äî
keep `(frontend)` imports free of Payload admin code.

**Image strategy**
- `next/image` for all photos (CMS `Media` + static) ‚Äî WebP, responsive `srcset`
- `priority` on the hero image only
- Explicit `width`/`height` on every image (prevent CLS)
- Max hero: 1200px wide, ‚â§ 200KB compressed

**Font strategy**
- Self-hosted Montserrat + Poppins, `font-display: swap`, `latin` + `latin-ext` subset
- Preload Montserrat (heading) + Poppins Regular used above the fold

**JS strategy**
- No third-party scripts blocking render
- GA4: `strategy="afterInteractive"` via `next/script`
- Count-up: vanilla JS + IntersectionObserver ‚Äî no library
- Below-fold sections via `dynamic()` import

## Accessibility requirements

- WCAG AA minimum across all components
- All interactive elements keyboard-navigable with visible focus ring
- Color contrast ‚â• 4.5:1 body text; ‚â• 3:1 large text (gold-as-text only on dark ‚Äî `design.md` ¬ß3.3)
- No information conveyed by color alone
- `prefers-reduced-motion` respected on all animations
- Form errors announced via `aria-live="polite"`, fields linked with `aria-describedby`
- Skip-to-content link as the first focusable element in the DOM
- Payload `Media` `alt` field is required ‚Äî no image can be saved without alt text
